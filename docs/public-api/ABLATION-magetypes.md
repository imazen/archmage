## Special Context (restated from task spec)

magetypes is the org's SIMD vector type library. The surface is **intentionally** dominated by:

- **Generic SIMD primitive types** (`u8x16`, `i16x8`, `f32x4`, `f32x8`, etc.) generated for every
  supported element type × vector width × token tier. Each type gets operators, conversions,
  lane-level accessors, comparison results, and masking support. The matrix IS the product.
- **Per-tier variants** (`u8x16` has `X64V3u8x16`, `X64V4u8x16`, `NeonU8x16`, `Wasm128U8x16`,
  `ScalarU8x16`, each implementing the common trait + natural operators). This is the
  design-by-tier dispatch model.
- **`WidthDispatch` trait** — auto-generated; provides all SIMD sizes from any token.
- **`#[magetypes]` macro output** — expands to the per-tier implementations, each carrying the
  full method matrix.

**None of the above should be flagged.** The investigation focused on the narrow complement:
hand-written items outside the deliberate design, particularly any stubs or placeholders
from intermediate development that were never cleaned up.

## Generated-Surface Attestation

The following categories were verified as macro-generated-by-design and are **out of scope**:

- **SIMD type impls** (`magetypes::simd::*`): generated by `xtask/src/simd_gen.rs` from
  `magetypes/type-registry.toml`. For each element type × width pair, a token-generic trait +
  per-tier concrete types are emitted. Spot-checked `magetypes/src/simd/generic/generated/`
  (e.g. `f32x8_impl.rs` — 2,341 lines, entirely macro-output, confirmed against xtask source).
  The pattern is identical across `u8x16`, `i16x8`, `u32x4`, `f32x4`, `f32x8`, etc.

- **Token backend trait impls** (`magetypes::archmage::tokens` re-exports): pulled from archmage.
  All `SimdToken` impls, `WidthDispatch` impls, tier-specific trait impls, and inherent impls on
  token structs are generated from `token-registry.toml` and are out of scope.

- **`WidthDispatch` trait**: auto-generated by `xtask`; shipped feature per CHANGELOG; KEEP.

- **`SimdTypes` trait** (`magetypes::types`): hand-written but intentional — provides associated
  types (`U8x16`, `I16x8`, …) for use in `#[magetypes]` generic programming. Verified in
  `magetypes/src/types.rs`. KEEP.

- **`#[doc(hidden)]` modules**: `nostd_math` and `bypass_adversarial` are already hidden;
  `nostd_math` is consumed by `zenpixels-convert` tests but correctly kept unexported.

Verified generated fraction: ≥ 99.4 % (≥ 24,640 / 24,798 all-features items).
The ~158 hand-written items are: `cast` module (3 items), `SimdTypes` trait + impls (~20 items),
`WidthDispatch` trait + impls (~30 items), `F32x8FromHalves`/`F32x16FromHalves` traits (~8 items),
prelude re-exports, `cast` cross-width utilities, and root module boilerplate.

## Module: `magetypes::cast` — vestigial placeholder traits

`magetypes/src/cast.rs` defines two traits and exports them publicly:

```rust
/// Downcast a SIMD vector to a narrower type.
pub trait Downcast<T> {}

/// Upcast a SIMD vector to a wider type.
pub trait Upcast<T> {}
```

**Analysis:**

These traits were added in commit `0d97db9` as "marker traits when `#[magetypes]` was implemented."
The module's own test (`cast.rs:52-60`) says:

> "This test just verifies the module compiles and documents the casting rules. Actual casting
> uses `.low()`, `.high()`, and `from_halves()`."

No implementations of `Downcast` or `Upcast` exist anywhere in the codebase — they are
marker interfaces with zero implementors. The actual API the crate ships for casting is:

- `.low()` / `.high()` — split a wider vector into narrower halves (generated per-type)
- `.from_halves(lo, hi)` — combine narrower into wider (generated per-type)
- `F32x8FromHalves` / `F32x16FromHalves` — hand-written cross-width builder traits

**External consumer scan (as of this scan):**
- `Downcast`: **0 external files** across all zen crates
- `Upcast`: **0 external files** across all zen crates
- The `cast` module path `magetypes::cast`: **0 external files** importing it
- Not mentioned in `docs/site/` (any `.md` or example file)

**Verdict: NOT FLAGGED — surfaced for awareness only.**

These traits are genuinely vestigial — zero implementors, zero consumers, documented in their
own test as "not the actual casting API." They are strong candidates for `#[doc(hidden)]` or
removal when a breaking change window opens.

However, **this report's conservative bar** is to propose changes only where the evidence is
overwhelming AND the cost of being wrong is low. For `Downcast`/`Upcast`, the cost of marking
them hidden or removing them is near-zero (zero consumers) — but magetypes is a load-bearing
org crate at 0.9.x, and these traits are also zero-cost to keep (no confusion risk if the actual
API is clearly documented elsewhere). On balance, **this falls below the flag threshold for an
immediate action item** but is the highest-confidence cleanup candidate in the crate.

If the team plans any breaking change in magetypes for another reason, removing `Downcast` and
`Upcast` should be bundled with that release (zero cost, zero consumer churn).

## Top Finding

**None.** The magetypes public surface is clean by design. The `cast` module's `Downcast`/`Upcast`

## Analyst Note on Scale

The 24,798-item surface is intentional and correct. A fully swept spot-check of the generated
output (f32x8, u8x16, WidthDispatch impls for all six tokens) confirmed each item follows the
`<TierToken><Width>` naming convention or the trait method matrix pattern. There are no orphan
items outside these families in the snapshot. The hand-written layer (~158 items) is clean.
