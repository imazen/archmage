## Special Context (restated from task spec)

archmage is the org's SIMD dispatch foundation. The surface is intentionally dominated by:

- **Capability tokens** (`X64V3Token`, `NeonToken`, etc.) and their trait impls — all generated
  from `token-registry.toml` via `cargo xtask generate`. Every token struct, `SimdToken` impl,
  tier-trait impl, `IntoConcreteToken` impl, and `dangerously_disable_token_process_wide` belongs
  to the deliberate token safety model.
- **Intrinsics re-exports** (`archmage::intrinsics::x86_64::*`) — generated by `cargo xtask generate`
  combining `core::arch` + `safe_unaligned_simd` memory ops in one namespace.
- **`#[arcane]`/`#[rite]`/`incant!` macro machinery** — required for the token/target_feature design.
- **`testing` module** (`for_each_token_permutation`, etc.) — actively used by 251 call sites
  across zen crates (jxl-encoder, zenresize, linear-srgb, zenwebp, zenmetrics, …).

**None of the above should be flagged.** The investigation focused on the narrow complement:
hand-written items outside the deliberate design.

## Generated-Surface Attestation

The following categories were verified as macro-generated-by-design and are **out of scope**
for this ablation:

- **Token structs and impls** (`archmage::tokens::X64V1Token` … `Avx512Fp16Token` and every
  ARM/WASM token): generated from `token-registry.toml` via `xtask/src/token_gen.rs`. Each
  token gets `SimdToken`, `IntoConcreteToken`, `HasX64V*/HasArm64V*/HasNeon*` trait impls,
  disable/enable/cache atomics, `__ARCHMAGE_TIER_TAG` (already `#[doc(hidden)]`), and
  arch-gated stub variants. Verified via `src/tokens/generated/x86.rs`,
  `src/tokens/generated/arm.rs`, `src/tokens/generated/wasm.rs`, and their `*_stubs.rs`
  counterparts.

- **Intrinsics** (`archmage::intrinsics::x86_64::*` et al.): generated by
  `xtask/src/intrinsics_gen.rs`. Each is a re-export of `core::arch` value-intrinsics plus
  explicit safe reference-based memory ops from `safe_unaligned_simd` that shadow the
  pointer-based versions. Not flagged.

- **`__ARCHMAGE_TIER_TAG` inherent consts**: all already `#[doc(hidden)]`. Verified in
  `src/tokens/generated/x86.rs:2960-2995`.

- **`__private` module**: already `#[doc(hidden)]`. Contains `assert_archmage_token` —
  the doc comment describes its use by `#[arcane]` macros, but the actual macro output
  (verified in `tests/expand/arcane/*.expanded.rs`) now uses a direct `const` array-index
  assertion against `__ARCHMAGE_TIER_TAG` instead. The module is inert but correctly hidden.

Verified generated fraction: ~90.7 % (3,381 / 3,726 all-features items).
The remaining ~345 items are hand-written (detect module, testing module, core trait API,
prelude, macros).

## Module: `archmage::detect` — assembly-inspection helpers

These five functions are labeled "helper function for verifying assembly output" in their
docstrings (`src/detect.rs:757-789`). They are `#[inline(never)]` and exist so users can
inspect the emitted assembly to confirm compile-time optimization is working.

**External consumer scan (as of this scan):**
- Org-wide grep: **0 files** in `/home/lilith/work/zen/` or any non-archmage org crate.
- Only consumers: `archmage`'s own `tests/token_infrastructure.rs:1062-1064` and
  `archmage-artisan/tests/token_infrastructure.rs:1059-1061`.
- Not mentioned in `docs/site/` (any `.md`) or the project `README.md`.

These are asm-inspection development tools, not part of the documented public API.
The crate already uses `#[doc(hidden)]` for `__priv_winarm` and `__ARCHMAGE_TIER_TAG`
for similar reasons. These should match that treatment.

| item | class | evidence | proposed action | semver impact |
|---|---|---|---|---|
| `pub fn archmage::detect::check_avx2_available() -> bool` | A | 0 external consumers (grep confirms); `tests/token_infrastructure.rs` comment: "asm-inspection helpers; just verify they return bools"; no docs/site mention | `#[doc(hidden)]` | None |
| `pub fn archmage::detect::check_fma_available() -> bool` | A | Same as above | `#[doc(hidden)]` | None |
| `pub fn archmage::detect::check_avx512f_available() -> bool` | A | Same as above | `#[doc(hidden)]` | None |
| `pub fn archmage::detect::check_sve_available() -> bool` | A | Same; aarch64-gated; 0 external consumers | `#[doc(hidden)]` | None |
| `pub fn archmage::detect::check_sve2_available() -> bool` | A | Same; aarch64-gated; 0 external consumers | `#[doc(hidden)]` | None |

**Note:** All five `check_*` functions share a single finding slot in the top-findings count
since they are a uniform group with the same disposition. Total: **5 items, all class A**.

**Proposed action:** Add `#[doc(hidden)]` to all five. These remain `pub` so the test code
can still call them without `#[allow(dead_code)]` complications, but they disappear from
`rustdoc` output and the public-API snapshot. When the snapshot regenerates, they should
also be excluded from `archmage.txt`.

## Module: `archmage::detect` — macros

`is_x86_feature_available!` and `is_aarch64_feature_available!` are `#[macro_export]`
public macros with full documentation in `src/detect.rs` explaining their use and
limitations. The crate's CLAUDE.md explicitly says "Use tokens instead of this macro" —
which is the recommended pattern — but the macros remain useful for non-generic detection
code outside of SIMD kernels.

**External consumer scan (as of this scan):**
- Org-wide grep: **0 external consumers** found.
- Macros are documented with examples in their docstrings.
- Not mentioned in `docs/site/` but the docstrings are the canonical docs for them.

**Verdict: KEEP.** The macros serve a documented purpose (compile-time-optimized feature
detection before entering the token system). Zero external consumers is consistent with
their role as an opt-in tool. The `__impl_*` implementation helper macros are already
correctly excluded from the public snapshot via the `_*` prefix filter.
