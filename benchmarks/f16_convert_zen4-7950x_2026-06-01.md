# f16 ↔ f32 slice-conversion: AVX-512F 16-wide vs F16C 8-wide (Zen 4 / 7950X)

Measures the AVX-512F 16-wide f16 path (`_mm512_cvtph_ps` / `_mm512_cvtps_ph`)
added in `convert_f16.rs` against the existing F16C 8-wide path
(`_mm256_cvtph_ps` / `_mm256_cvtps_ph`) and the branchless scalar software kernel.

## TL;DR

On Zen 4 the 16-wide path is a **modest** win over 8-wide F16C, **not 2×**:

- **Compute-bound (L1-resident, ~4 K elems):** decode **≈1.2–1.5×**, encode **≈1.5–1.8×**.
- **Memory-bound (≥1 M elems):** decode **≈1.0–1.2×**, encode **≈1.0×** (parity).
- **Tiny (n=16):** dispatch-dominated; 16-wide is at parity / marginally slower.

**Why not 2×:** Zen 4 executes AVX-512 on **double-pumped 256-bit units** — a
512-bit `vcvtph2ps`/`vcvtps2ph` is internally 2×256-bit, the same convert-unit
throughput as two F16C ops. The win that *does* show up comes only from the
reduced loop/instruction count (half the iterations, one 512-bit load/store
instead of two 256-bit), which helps most when L1-resident and the loop
overhead is a larger fraction. Once memory bandwidth is the limit, both widths
saturate the load/store path and the advantage disappears.

**Not measured here:** Intel parts with a *native* 512-bit datapath (Skylake-X,
Ice Lake, Sapphire Rapids server) should show a larger gap because the convert
unit is genuinely 512-bit wide there. This box (Zen 4) cannot measure that — do
**not** extrapolate the Intel number from these results.

The change remains worth keeping regardless: it is **bit-identical** to the
8-wide path (verified exhaustively), never a regression on Zen 4, and the
modest Zen-4 win is free (same instructions, fewer of them).

## Hardware / build

- **Box**: AMD Ryzen 9 7950X (Zen 4), water-cooled, 128 GB. `/proc/cpuinfo`:
  `avx512f avx512bw avx512cd avx512dq avx512vl avx512_bf16 … f16c` —
  **no `avx512_fp16`** (Zen 4 lacks it; that is Zen 5 / Sapphire Rapids).
- **rustc**: 1.96.0. **`RUSTFLAGS=""`** (no `target-cpu=native`).
- archmage commit: `9c63dc2` (the V4 f16 path) + this bench.
- All runs `taskset -c 0` (pin one core, kill migration/turbo-core noise).

## Method

Two complementary harnesses:

1. **Clean single-process interleaved microbench** (headline, the 8-vs-16 width
   isolation). Both kernels in one binary, called A/B/A/B on the same buffer so
   they share identical ambient conditions; `min` over reps captures best-case
   throughput (filters scheduler noise). This isolates the *instruction-throughput*
   question. Source inlined below (one-shot; raw `std::arch` intrinsics — same
   instructions the production kernels emit). The production bench (#2) can't pin
   8-wide on a V4 box because its slice path summons-up, so this microbench is the
   authoritative 8-vs-16 source.

2. **Production dispatch** (`magetypes/benches/f16_convert.rs`, zenbench,
   `--features "std avx512"`, one process). `scalar` (software) and `f16c` =
   `X64V3Token` slice (the production path — summons-up to the best tier, so
   16-wide on this V4 box). Answers HW-vs-software through the full dispatch.

## Results — clean microbench (16-wide speedup over 8-wide F16C, `min` ns/elem)

| n | working set | decode 16w/8w | encode 16w/8w | regime |
|--:|--:|--:|--:|:--|
| 1 024 | 6 KB | 1.00× | 1.50× | L1, loop-overhead/timer-limited |
| 4 096 | 24 KB | **1.23–1.50×** | **1.43–1.82×** | **L1, compute-bound (cleanest)** |
| 65 536 | 384 KB | 1.04× | 1.30× | L2 |
| 1 048 576 | 6 MB | 1.21× | 1.08× | ~L3 / memory |
| 8 388 608 | 48 MB | 1.15× | 1.00× | DRAM-bound |

(decode at n=4096 measured 1.23× and 1.50× on two harness variants — real
run-to-run spread; reported as a range, not a single digit. The 4 K row is the
most reliable: per-call time is well above timer resolution and the buffer is
L1-resident so the result is compute-bound rather than bandwidth-bound.)

## Results — production dispatch (zenbench, `--features "std avx512"`)

Same process. `f16c` is the `X64V3Token` slice (summons-up to best → 16-wide here).

| op | n | scalar | f16c (V3↑) | hw/scal |
|:--|--:|--:|--:|--:|
| decode | 256 | 155 ns | 9.0 ns | 18× |
| decode | 4 096 | 2.40 µs | 114 ns | 21× |
| decode | 65 536 | 39.0 µs | 2.70 µs | 14× |
| decode | 1 048 576 | 623 µs | 53.9 µs | 12× |
| encode | 256 | 118 ns | 8.6 ns | 14× |
| encode | 4 096 | 1.80 µs | 113 ns | 16× |
| encode | 65 536 | 29.6 µs | 2.60 µs | 11× |
| encode | 1 048 576 | 474 µs | 47.8 µs | 10× |

HW is 10–21× over scalar (n=16 omitted: per-call time near timer resolution). The
once-per-slice summon is amortized to negligible — when measured against a direct
`X64V4Token` slice (before that redundant impl was removed) the two were within
1% at every size. The 8-wide-vs-16-wide width win is the microbench above (~1.2×
decode / ~1.9× encode compute-bound, ~1.1× memory-bound).

## Reproduce

Production dispatch (one build):
```
taskset -c 0 cargo bench -p magetypes --bench f16_convert --features "std avx512"
```

Clean microbench (one-shot; saved as inline source, not a committed crate):
```rust
// rustc -O -C target-cpu=x86-64-v3 f16ab.rs -o f16ab && taskset -c 0 ./f16ab
#![allow(internal_features)]
use std::arch::x86_64::*;
use std::time::Instant;
#[target_feature(enable = "f16c,avx,avx2")]
unsafe fn dec8(inp:&[u16],out:&mut[f32]){let n=inp.len();let mut i=0;while i+8<=n{let h=_mm_loadu_si128(inp.as_ptr().add(i) as *const __m128i);let f=_mm256_cvtph_ps(h);_mm256_storeu_ps(out.as_mut_ptr().add(i),f);i+=8;}}
#[target_feature(enable = "avx512f")]
unsafe fn dec16(inp:&[u16],out:&mut[f32]){let n=inp.len();let mut i=0;while i+16<=n{let h=_mm256_loadu_si256(inp.as_ptr().add(i) as *const __m256i);let f=_mm512_cvtph_ps(h);_mm512_storeu_ps(out.as_mut_ptr().add(i),f);i+=16;}}
#[target_feature(enable = "f16c,avx,avx2")]
unsafe fn enc8(inp:&[f32],out:&mut[u16]){let n=inp.len();let mut i=0;while i+8<=n{let f=_mm256_loadu_ps(inp.as_ptr().add(i));let h=_mm256_cvtps_ph::<0>(f);_mm_storeu_si128(out.as_mut_ptr().add(i) as *mut __m128i,h);i+=8;}}
#[target_feature(enable = "avx512f")]
unsafe fn enc16(inp:&[f32],out:&mut[u16]){let n=inp.len();let mut i=0;while i+16<=n{let f=_mm512_loadu_ps(inp.as_ptr().add(i));let h=_mm512_cvtps_ph::<0>(f);_mm256_storeu_si256(out.as_mut_ptr().add(i) as *mut __m256i,h);i+=16;}}
fn mn(v:&[f64])->f64{v.iter().cloned().fold(f64::INFINITY,f64::min)}
fn main(){
  assert!(is_x86_feature_detected!("avx512f")&&is_x86_feature_detected!("f16c"));
  for &n in &[1024usize,4096,65536,1048576,8388608]{
    let i16:Vec<u16>=(0..n).map(|i|((i.wrapping_mul(2654435761))&0x7bff) as u16).collect();
    let mut o32=vec![0f32;n];
    let inf:Vec<f32>=(0..n).map(|i|(i as f32)*0.001).collect();
    let mut o16=vec![0u16;n];
    let reps=(200_000_000/n).max(50);
    for _ in 0..(reps/10+10){unsafe{dec8(&i16,&mut o32);dec16(&i16,&mut o32);enc8(&inf,&mut o16);enc16(&inf,&mut o16);}}
    let(mut d8,mut d16,mut e8,mut e16)=(vec![],vec![],vec![],vec![]);
    for _ in 0..reps{
      let t=Instant::now();unsafe{dec8(&i16,&mut o32)};d8.push(t.elapsed().as_nanos() as f64/n as f64);
      let t=Instant::now();unsafe{dec16(&i16,&mut o32)};d16.push(t.elapsed().as_nanos() as f64/n as f64);
      let t=Instant::now();unsafe{enc8(&inf,&mut o16)};e8.push(t.elapsed().as_nanos() as f64/n as f64);
      let t=Instant::now();unsafe{enc16(&inf,&mut o16)};e16.push(t.elapsed().as_nanos() as f64/n as f64);
    }
    println!("n={n:>8} dec {:.2}x enc {:.2}x", mn(&d8)/mn(&d16), mn(&e8)/mn(&e16));
  }
}
```
